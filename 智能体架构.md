# AETHER-DFT 智能体架构

> 本文件是 AETHER-DFT 的唯一根目录 canonical live doc：用于人类和智能体共同维护“当前共识方案、决策和待办索引”。
> 它不是执行态数据库；具体项目进展仍以 `.aether/projects/<slug>/_state.md` 与 `research_progress.md` 为准。

---

## 1. 一句话目标

AETHER-DFT 是一个计算化学 / DFT 领域的对话式科研合伙人：它通过持续对话理解课题、提出可执行方案、调用结构与计算工具，并把科研进展沉淀到 `.md` 项目状态中。

它的重点不是一次性端到端自动跑完，而是像同组科研伙伴一样逐步推进：讨论方案 → 准备结构 → 生成候选 → 再进入后续计算阶段。

---

## 2. 权威边界

### 2.1 本文件负责什么

`智能体架构.md` 只负责：

- 当前产品/智能体定位
- 共识方案与设计决策
- 主线流程索引
- 待实现能力清单
- 给后续智能体的轻引导说明

### 2.2 本文件不负责什么

本文件不保存执行态真相。以下信息以项目目录为准：

- 当前课题做到哪一步
- 当前卡点
- 已完成计算
- 下一步实际执行动作
- 计算记录与结果

执行态真源固定为：

```text
.aether/projects/<slug>/_state.md
.aether/projects/<slug>/research_progress.md
```

如果本文件与项目状态冲突，优先信任 `.aether/projects/<slug>/_state.md` 与 `research_progress.md`。

`README.md` 与 `AGENTS.md` 不在本 canonical live doc 契约内。

---

## 3. 轻引导原则

AETHER-DFT 采用“自由模式 + 轻引导”：

- 不给模型硬塞固定角色。
- 不限制模型只能说某一类内容。
- 只给当前阶段的方向提示，让模型自由判断最重要的科研信息。
- 不硬编码具体材料、吸附质、反应路径或课题。

核心原则：

> 不要告诉模型它必须是谁；只告诉它当前科研阶段需要注意什么。

## 4. 当前主线：先做前两步

### Step 1 — 课题讨论与方案沉淀

智能体应先和用户讨论课题：

- 研究对象是什么
- 科学问题是什么
- 可能的机理假设是什么
- 需要什么结构与计算证据
- 当前方案是否足以进入建模

讨论产生的共识可以沉淀到：

- 本文件的“设计决策 / 待办索引”
- 对应项目的 `_state.md` / `research_progress.md`

其中，执行态仍以后者为准。

### Step 2 — 结构获取、生成与操作

得到方案后，智能体应进入结构阶段：

1. 获取或读取结构文件
2. 做必要的结构检查
3. 建 slab / cluster / defect / vacancy / dopant
4. 添加吸附物或中间体
5. 生成候选构型
6. 把候选结构交给后续计算阶段

当前默认首选主线为：

```text
adsorption_plan -> adsorption_build_slab -> adsorption_candidates
```

其他结构工具只作为 supporting utilities。

## 5. 工具边界

### 5.1 第一阶段优先工具

优先支持：

- 结构导入与格式转换：`.xsd ⇄ POSCAR`、POSCAR/CONTCAR 读取
- 结构检查：键长、位移、异常接触、吸附物漂移
- slab / surface 建模
- 吸附物添加
- 候选构型生成

### 5.2 后置阶段

以下能力明确后置，不混入前两步主线：

- submit
- monitor
- fetch
- parse
- analyze
- export
- 真实集群提交

---

## 6. Prompt / Context 接入规则

本文件可以被 prompt/context 读取，但只能作为 volatile runtime context：

- 允许：短摘录 / digest / 路径提示
- 不允许：全文灌入 stable prefix
- 不允许：覆盖 project state
- 不允许：把本文件当作执行态真源

测试应直接断言：

- `compile_system_prompt` / `build_prompt_packet` 能读取本文件摘要
- 本文件只进入 `volatile_suffix_text` / `volatile_layer_names`
- 本文件不进入 `stable_prefix_text` / `stable_layer_names`

---

## 7. 当前设计决策

- 根目录唯一 canonical live doc：`智能体架构.md`
- 执行态真源：`.aether/projects/<slug>/_state.md` / `research_progress.md`
- 默认模型：`deepseek:deepseek-v4-pro`；内置只暴露 `deepseek:deepseek-v4-pro` 与 `bailian:qwen3.7-max` 两个适合本项目的 OpenAI-compatible 模型
- Codex 子智能体默认优先使用 `gpt-5.5`
- 不为了等子智能体而空转；应一边推进本地可做的事，一边等待
- 前两步主线先收束，不提前扩到提交/解析/监控
