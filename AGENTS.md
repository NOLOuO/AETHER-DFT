# AGENTS.md

## 一句话目标

构建一个**计算化学领域的对话式科研合伙人**——用户通过持续、多轮的对话，与智能体一起一步步推进并跟踪具体的科研项目，而不是一次性下指令、拿结果。

它不是通用智能体，也不是 Claude Code / Codex 那类「写代码、跑命令」的工程助手。它的本质是一个**懂计算化学的科研伙伴**。

---

## 核心定位：科研合伙人，不是工具

| | Claude Code / Codex（要差异化的对象） | 本智能体 |
|---|---|---|
| 服务对象 | 通用软件工程 | 计算化学 / DFT 科研 |
| 交互模式 | 给指令 → 执行任务 → 结束 | 持续对话 → 共同推进 → 长期跟踪 |
| 记忆 | 单次会话 / 仓库上下文 | 项目级科研状态的累积与延续 |
| 知识 | 通用编程 | 内化计算化学领域知识（机理、方法、参数选择、文献） |
| 角色 | 执行者 | 合伙人：会追问、会建议方向、会记得我们做到哪了 |

**关键：领域性是护城河。** 它理解的是反应机理、电子结构、催化体系、计算方法学，而不是泛泛的「帮我跑个脚本」。

---

## 核心使用场景（这是真正要达成的）

用户与智能体长期对话，智能体始终「在场」地参与一个具体科研项目，做到：

1. **持续推进** — 每次对话都基于上一次的进展往前走，不是从零开始。
2. **主动跟踪** — 记得当前项目做到哪一步、卡在哪、下一步该做什么。
3. **科研判断** — 能就机理、方法选择、参数设置、结果解读给出有领域依据的建议，而非机械执行。
4. **闭环落地** — 在对话中自然地把讨论转化为可执行的计算（建结构、生成输入、提交、回收结果、解读），再把结果带回对话继续讨论。

一句话：**像一个真实的、懂行的同组合作者，可以一直 chat 着把一个 DFT 课题推下去。**

---

## 实现这个目标所需的能力层（架构服务于上面的场景）

四层 + 贯穿各层的共享数据模型，全部是为了支撑「持续对话推进科研」这件事：

- **对话层** — 理解科研意图，维持多轮上下文，是合伙人「说话」的地方。
- **任务建模层** — 把对话中的科研意图转成结构化计算任务。
- **结构建模层** — 围绕 6 个核心结构对象 + 10 个初始结构工具，处理体系建模。
- **导出/执行层** — 生成计算输入、提交执行、回收结果。
- **共享数据模型** — 承载**项目级科研状态**，让进展能在多轮、多次会话之间累积与延续（这是「持续跟踪」能成立的关键）。

## 整合范围

统一现有三个项目的能力，服务于上述科研合伙人定位：

1. `literature-review` — 文献检索（合伙人的「读文献」能力）
2. `domestic-research-copilot-app` — 多模型并行分析（合伙人的「多角度思考」能力）
3. `semi_auto_dft` — planner → builder → runner → parser（合伙人的「动手算」能力）

## 不做什么（边界）

- 不做通用编程 / 通用任务智能体。
- 不与 Claude Code / Codex 在工程执行上竞争——领域科研判断与长期项目陪伴才是差异点。
